# Mochi 软件 MVP PRD（陪伴型核心体验）

> 目标：在 demo 阶段，验证 **Mochi 作为“情绪陪伴软件”的核心价值是否成立**，而不是验证硬件能力。 关键词：被接住｜关系连续性｜情绪可视化｜低负担表达

---

## 一、产品定位（MVP 版）

**Mochi 是什么？**\
一个帮助用户把「当下情绪」安全放下、被承接、并可继续回来的陪伴型软件。

**MVP 不做什么**

- 不做心理咨询
- 不做情绪诊断 / 打分
- 不追求“解决问题”

**MVP 核心价值一句话**

> 帮用户把说不清的情绪，先放下来，并且下次还能从这里继续。

---

## 二、核心用户画像（Demo 期）

- 25–35 岁
- 独居 / 情绪更多内化
- 有表达欲，但不想被分析或教育
- 对「记录」不排斥，但讨厌打卡与量化

**关键心理需求**

- 表达时不被打断、不被评判
- 情绪被“点破但不下结论”
- 对话不是一次性的

---

## 三、核心使用路径（MVP）

### 总览路径

```
首次打开
 → 轻 Onboarding
 → 发生一次对话（App / Mochi）
 → 对话结束
 → 情绪被存入「情绪罐头」
 → App 给出轻承接（summary / 下一步）
 → 下一次自然回访
```

---

## 四、Onboarding 设计

### 目标

- 30 秒内进入第一次有效对话
- 不建立心理防御

### 流程

**Step 1｜欢迎页（价值对齐）**

- 文案示例：
  > 这里不是心理咨询，也不是建议机器人。\
  > 你可以把一段情绪，先放在这里。

**Step 2｜立即表达（核心）**

- 输入方式：文字 / 语音（二选一）
- Placeholder：
  > 你现在的感受是……（不用整理）

**Step 3｜首次回应（关键）**

- Mochi 第一条回复必须做到：
  - 复述情绪
  - 不下判断
  - 不给建议

---

## 五、对话模块（核心功能｜MVP 示意｜聊天页为主）

> Demo / Vibe coding 阶段：**对话页面为标准聊天形态**，用于承载用户与 Mochi（娃娃 / App）的真实交流细节； 情绪结构化（blob / jar）在后台自动完成，不在聊天页显性呈现。

### 5.1 对话页定位

- Chat 页面 = **表达发生的地方（过程态）**
- Home 页面 = **当天状态的承接与总览（结果态）**

Chat 页面需要：

- 完整聊天历史（用户 ↔ Mochi）
- 支持连续、多次会话
- 不展示情绪罐头 / blob 结构

---

### 5.2 对话结构（示意｜无模型）

一次对话以「自然聊天流」进行：

1️⃣ 用户输入（文字 / 语音）\
2️⃣ Mochi 占位回应（复述 / 陪伴式）\
3️⃣ 聊天自然往复（不强行推进）

> Demo 阶段不要求对话智能度，只需保证：
>
> - 有来有回
> - 语气温和
> - 不下判断、不解决问题

---

### 5.3 Session 结束判定（规则示意）

系统通过**时间与交互规则**判断一次对话 session 结束：

- 用户最后一条消息后 **N 分钟无新输入**（如 5–10 分钟）
- 或用户主动关闭 App

一旦判定 session 结束：

- 后台生成 EmotionBlob（用于 Home / Jar）
- Chat 页面准备展示 Session End Card

---



### 5.2 对话结构（示意）

一次对话由以下阶段组成：

1️⃣ 用户输入（文字 / 语音）\
2️⃣ 系统回应（占位）：

- 复述用户情绪
- 不给建议

示例占位回应：

> 听起来这件事让你有点难受，我在。

3️⃣ 可选追问（规则驱动）：

- “这件事发生在什么时候？”
- “当时你最明显的感觉是什么？”

---

### 5.3 情绪碎片生成规则

当满足以下任一条件，即生成一个 emotion blob：

- 用户明确表达感受
- 用户描述具体事件
- 对话超过 N 轮

blob 写入：

- 时间
- 用户原话 / 摘要
- 情绪关键词（rule-based）

---



## 六、情绪罐头（Emotion Jar｜按「一天」聚合）

### 6.1 定位（修正）

**一个情绪罐头 = 用户的一天，而不是一次对话。**\
它是「这一天里所有情绪碎片的容器」。

- 对话 ≠ 罐头
- 罐头 ≠ 聊天记录
- 罐头是时间维度的情绪承载

---

### 6.2 情绪碎片（Emotion Blob）

**一次对话 = 产生多个情绪碎片（blob）**，而不是生成罐头。

每个情绪碎片包含：

- 时间点
- 触发事件（可选）
- 用户原话（或摘要）
- 当前情绪状态（模糊词即可）

多个 blob 在同一天内，被自动归入同一个情绪罐头。

---

### 6.3 罐头生成与更新逻辑

- 每天首次产生情绪碎片 → 创建当天罐头
- 当天后续对话 → 持续往罐头中追加 blob
- 用户无需感知「生成动作」，只看到罐头逐渐被填充

---

### 6.4 用户可见形态（MVP｜含历史入口）

**首页 Home 为唯一主入口，不新增复杂页面层级。**

- 首页主区域：**今日情绪罐头（Today Jar）**
  - 以“今天”为时间锚点
  - 展示当日整体情绪状态（非明细）

- 首页次入口：**历史罐头入口（History Entry）**
  - 形式：
    - 顶部/右上角「过去的日子」按钮，或
    - 今日罐头下方「查看之前的日子」轻入口
  - 语义是“回看”，而不是“管理”

点击历史入口后：
- 进入 **历史罐头列表页（History Jars List）**
- 按日期倒序展示 DayJar（最近优先）
- 每个 DayJar 只展示：
  - 日期
  - 一句话状态摘要（DayReflection 或占位文案）

不展示：
- 情绪碎片明细
- 统计图 / 数量
- 任何操作按钮

---

### 6.5 历史罐头的交互原则（MVP｜可查看细节）

**历史罐头需要“可回看细节”，但不引入复杂分析或二级结构。**

核心原则：
- 罐头 = 一天
- Blob = 当天留下的真实记录
- Blob 的来源可以不同，但在视觉上被统一承载

---

### 6.6 情绪碎片（Blob）的来源模型

每一个 EmotionBlob 必须携带来源信息（source），但对用户保持低感知。

可能来源包括：
- `chat`：与 Mochi / App 的对话
- `manual`：用户手动记录的一句话
- `ring_voice`：戒指录音转写摘要
- `device_other`：其他硬件来源（预留）

```ts
EmotionBlob {
  id: string
  timestamp: Date
  source: 'chat' | 'manual' | 'ring_voice' | 'device_other'
  rawText: string
  emotionHint: string
  sessionId?: string
}
```

---

### 6.7 历史罐头查看方式（MVP）

点击历史罐头后：

- 进入 **同一套罐头视图（与今日一致）**
- 展示该日的 EmotionBlob 时间线
- Blob 以“记录片段”形式呈现，而非聊天记录

Blob 展示建议：
- 时间点 + 一句话内容
- 极轻的来源提示（icon / tag，而非文字说明）

示例：
> 10:30｜有点乱（💬）
> 15:10｜不想回消息（📝）
> 21:40｜录音：下班路上很空（🎧）

---

### 6.8 从历史 Blob 继续的交互

点击任一历史 Blob：

- 默认行为：
  - 进入 Chat 页面
  - 顶部显示极轻上下文提示（一次性）：

示例：
> 这是你在 6 月 12 日留下的一段记录。

- 随后开启 **新的 Session**：
  - 新产生的内容写入“今天”的 DayJar
  - 原历史罐头内容保持不变

---







## 七、对话结束后的承接机制（Session End Card｜核心）

> Session End Card 是 **聊天流中的最后一条“轻量消息”**，不是弹窗、不是独立页面。 它的职责是：为这段聊天画一个温柔的句号。

---

### 7.1 出现时机（必须克制）

Session End Card **不在每次对话后出现**，仅在满足以下任一条件时触发：

- 对话轮次 ≥ N（如 3 轮）
- 用户明确表达情绪 / 消耗感
- 对话持续时间较长

否则：

- 聊天自然结束
- 不出现卡片

---

### 7.2 卡片内容结构（固定三段）

**① 承接句（确认被接住）**

- 复述刚刚的对话重点
- 不抽象、不评判

示例：

> 刚才你说的那段，其实挺消耗的。

**② Soft Next Step（唯一一条，可忽略）**

- 情绪向 / 身体向
- 可来自用户刚刚明确提到的具体事物（如书、音乐）

示例：

> 如果你愿意，今晚可以翻翻你刚才提到的那本《蛤蟆先生的心理学》，不用读完。

或：

> 现在不想继续想也没关系，去做一件能让身体放松的小事就好。

**③ 结束语（隐含关系连续性）**

- 不强调下次
- 不做提醒

示例：

> 等你回来，我们可以从这里继续。

---

### 7.3 交互方式

- 作为聊天流最后一条消息展示
- 样式：卡片感 / 低对比度
- 用户无需点击任何按钮
- 向上继续聊天 / 向下滑离开均可

卡片内容 **不存档、不回看**：

- 情绪已通过 EmotionBlob 进入 Home
- Next Step 为短暂态，不形成任务列表

---



### 7.2 当天结束承接（日级）

**发生在：一天即将结束 / 用户回到 App**

系统基于当天罐头生成：

- 一段「今天的状态回看」
- 不做总结判断，只做状态呈现

示例：

> 今天你有几次明显的情绪起伏， 多数都和「被忽视」有关。 这些都已经被放进今天的罐头里了。

---

### 7.3 跨天承接（关系连续性）

当用户在未来某天打开 App：

- 默认从「最近一个罐头」进入
- 轻提示关系承接，而不是提醒打卡

示例：

> 前几天你提到的那种堵，其实还没完全走。 如果你愿意，我们可以从那里继续。

---



## 八、硬件在 MVP 中的角色

### 原则

- **硬件不是价值来源**
- 只是「低负担表达入口」

### 支持能力

- 语音输入
- 聊天触发
- 情绪记录入口

**所有价值沉淀在软件中**

---

## 九、MVP 成功判断指标（Demo）

- 是否在首次使用中完成一次完整对话
- 是否生成至少 1 个情绪罐头
- 是否在 48 小时内回访
- 是否主动点击罐头继续聊

---

## 十、明确不做清单（防跑偏）

- 不做复杂情绪分析
- 不做量表 / 分数
- 不强行推动深度
- 不做“你应该”式引导

---

## 十一、一句话收尾

> Mochi MVP 的成功，不在于用户说了多少， 而在于他 **愿不愿意再回来，从上一次没说完的地方继续。**

---

## 十二、MVP 状态结构（用于 Vibe Coding / 前端实现）

> 本章节不是 PRD 描述，而是**给前端 / vibe code 用的状态模型**，确保不会被实现成「聊天 App」。

---

## 12.1 核心状态对象（Data Model）

### ① DayJar（情绪罐头｜一天）

```ts
DayJar {
  id: string            // yyyy-mm-dd
  date: Date
  blobs: EmotionBlob[]  // 当天所有情绪碎片
  dayReflection?: DayReflection
}
```

**语义说明**

- 一个 DayJar = 用户的一天
- 不主动创建，由第一个 EmotionBlob 自动触发
- 用户不感知“创建动作”

---

### ② EmotionBlob（情绪碎片）

```ts
EmotionBlob {
  id: string
  timestamp: Date
  source: 'chat' | 'voice'
  rawText: string       // 用户原话或转写
  emotionHint: string   // 模糊情绪词："乱" "堵" "累"
  sessionId: string
}
```

**语义说明**

- 一个 Blob ≠ 一次完整对话
- 它只是一个「被表达的瞬间」
- 多个 blob 组成一天

---

### ③ Session（一次对话）

```ts
Session {
  id: string
  startAt: Date
  endAt?: Date
  blobs: EmotionBlob[]
  reflection?: SessionReflection
  nextStep?: SoftNextStep
}
```

**语义说明**

- Session 是临时态
- 结束后只留下 reflection / nextStep
- 不作为长期核心对象

---

### ④ SessionReflection（对话即时小结）

```ts
SessionReflection {
  bullets: string[]   // 1–3 条
}
```

**规则**

- 只复述，不判断
- 语言贴近用户原话

---

### ⑤ SoftNextStep（轻量下一步）

```ts
SoftNextStep {
  suggestion: string
  accepted?: boolean  // 可忽略
}
```

**规则**

- 情绪向，不是任务
- 不追踪完成率

---

### ⑥ DayReflection（一天回看）

```ts
DayReflection {
  summary: string
}
```

**规则**

- 只在一天将结束时生成
- 不出现 to-do
- 不出现性格判断

---

## 12.2 页面与状态流转

### 首页 Home

- 当前 DayJar（今日）
- 历史 DayJar（按日期）

---

### DayJar 详情页

- 情绪碎片时间线（EmotionBlob list）
- 点击某一 blob → 继续聊（新 Session）

---

### 对话页 Chat

状态流：

```
Idle
 → UserInput
 → SystemReply (占位)
 → [生成 EmotionBlob]
 → 可选追问
 → Session End
 → Reflection + SoftNextStep
```

---

## 12.3 Blob 生成时机（MVP 规则）

生成 EmotionBlob 当且仅当：

- 用户表达感受（出现情绪词）
- 或描述具体事件
- 或一次对话超过 N 轮（如 3 轮）

---

## 12.4 Demo 阶段刻意不做的事

- 不做情绪统计图
- 不做连续天趋势
- 不做情绪评分
- 不做人格分析

---

## 12.5 判断 MVP 是否「跑通」的信号

- 用户一天内产生 ≥2 个 EmotionBlob
- 用户主动点开历史 blob 继续聊
- 第二天打开 App 查看前一天罐头

---

> **如果用户愿意把一天放进罐头里， 并在第二天回来打开它， 这个陪伴产品就已经成立了。**
